## 2024-05-24 - React.memo for Streaming Chat Interfaces
**Learning:** In a streaming chat interface, new chunks of text trigger re-renders of the entire message list if state is held in a parent component (like App). This can cause severe performance degradation (O(n^2) rendering cost) as the chat gets longer because every previous message and its complex Markdown components re-render on every tiny chunk of text received.
**Action:** Always wrap chat message components and expensive inner components (like Markdown renderers) in `React.memo` to prevent re-rendering historical messages while the active message is streaming. Additionally, ensure complex objects passed as props (like markdown components maps) are hoisted outside the render cycle so they maintain referential equality.

## 2024-05-18 - Batching React State Updates in SSE Streams
**Learning:** In React, calling a state updater function (like `setMessages`) inside a tight `for` loop parsing individual Server-Sent Event (SSE) lines from a single stream chunk causes excessive state updater executions and array cloning. Even with React 18's automatic batching, executing the updater function and spreading the array for every line is highly inefficient, especially when a single stream read yields multiple lines (e.g., when the frontend receives data slightly delayed or chunked together by the network).
**Action:** Always accumulate parsed data (e.g., string concatenation) within the scope of a single network read or event payload, and call the React state updater only once with the final accumulated data to minimize expensive state recalculations and intermediate object creation.
